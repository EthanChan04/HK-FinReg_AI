### 3.2 核心功能详解

#### 3.2.1 SVF合规审查模块（RAG模式）

**功能描述：**
基于HKMA《Guideline on AML/CFT for SVF Licensees》，对储值支付工具的交易和客户资料进行合规审查。

**AI技术应用：**
- **RAG检索增强生成**：将96页HKMA监管指南向量化存储，实时检索相关条款
- **语义搜索**：理解用户查询意图，找到最相关的监管内容
- **结构化输出**：生成标准化的风险评估报告

**关键Prompt设计（我的决策）：**

```python
prompt = """
You are a Senior Compliance Officer for a Hong Kong FinTech/SVF Company. 
Your task is to review the following transaction or customer profile 
against the HKMA guidelines provided in your context.

### Guidelines for Analysis:
1. **Check Thresholds**: specifically look for SVF stored value limits
2. **Verify Identity Type**: Distinguish between Device-based vs Network-based
3. **Avoid Repetition**: Keep professional and concise

### Output Structure:
1. **Risk Level**: (Low / Medium / High)
2. **Regulatory Analysis**: Cite specific chapters/paragraphs
3. **Recommendation**: (Approve / Reject / Request Enhanced Due Diligence)
"""
```

**决策思考过程：**
- 为什么用RAG而不是Fine-tuning？监管条例频繁更新，RAG更容易维护
- 为什么选择这种输出结构？合规人员需要结构化的决策依据
- 为什么要求引用具体章节？增强可信度，符合审计要求

**功能截图/说明：**
输入：交易记录或客户KYC信息
输出：
1. 风险等级（低/中/高）
2. 监管分析（引用HKMA具体条款）
3. 建议（批准/拒绝/要求强化尽职调查）

#### 3.2.2 虚拟银行开户审查模块（LLM-only模式）

**功能描述：**
针对虚拟银行开户场景，进行身份验证评估、CDD级别判断、风险分类等。

**AI技术应用：**
- **LLM内置知识**：利用大模型已有的香港金融监管知识
- **结构化推理**：按标准流程逐步评估开户资格
- **实时决策**：无需额外文档加载，响应速度快

**我的产品决策：**
- 为什么用LLM-only而不是RAG？没有现成的虚拟银行开户PDF，但LLM已有相关知识
- 后续如何优化？可以添加HKMA《虚拟银行的认可》指南到知识库

**输出结构：**
1. 申请摘要
2. 身份验证评估
3. CDD级别评估
4. 风险分类
5. 监管关注
6. 建议

#### 3.2.3 跨境汇款评估模块（LLM-only模式）

**功能描述：**
对跨境汇款交易进行风险评分、制裁筛查、AML合规检查等。

**对标Fusion Bank场景：**
- 直接关联Fusion Bank的"全球汇款"业务
- 评估框架符合HKMA对金融机构的跨境合规要求

**AI技术应用：**
- **风险评分系统**：1-10分的量化风险评估
- **多维度分析**：国家风险、交易风险、客户风险
- **制裁筛查**：模拟多制裁名单检查

#### 3.2.4 SME融资评估模块（LLM-only模式）

**功能描述：**
对中小企业融资申请进行信用评级、业务可行性分析、风险调整定价等。

**对标Fusion Bank场景：**
- 直接关联Fusion Bank的"电商宜享贷"产品
- 针对跨境电商企业的专门评估框架

**AI技术应用：**
- **信用评级**：A-E五级评级体系
- **替代数据利用**：电商平台数据、交易历史等
- **风险调整定价**：基于风险的利率建议

#### 3.2.5 PII数据脱敏模块

**功能描述：**
在发送数据给大模型之前，自动识别并脱敏敏感个人信息。

**AI技术应用：**
- **正则表达式匹配**：精准识别香港身份证、手机号格式
- **实时处理**：在调用LLM前即时脱敏
- **可逆设计**：内部系统保留原文，外部只传脱敏数据

**我的产品决策：**
- 为什么需要这个？符合香港《个人资料（私隐）条例》（PDPO）要求
- 为什么用正则而不是AI？简单场景正则更可靠、更快、更便宜

**脱敏规则：**
- 香港身份证：`A123456(7)` → `[HKID REDACTED]`
- 香港手机号：`91234567` → `[PHONE REDACTED]`

#### 3.2.6 性能实时监控模块

**功能描述：**
实时追踪系统性能指标，包括查询次数、响应时间、错误率等。

**AI技术应用：**
- **会话级追踪**：每个用户会话独立统计
- **模块级细分**：分模块统计使用情况和性能
- **实时更新**：每次查询后立即更新统计数据

**监控指标：**
- 总查询次数
- 平均响应时间
- 模块使用统计
- 错误次数

### 3.3 关键技术决策（我的思考过程）

#### 3.3.1 双模架构设计

**问题：** 如何平衡准确性和灵活性？

**我的分析：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **纯RAG** | 准确性高，可追溯 | 新场景需要准备文档，迭代慢 |
| **纯LLM-only** | 灵活快速，易于扩展 | 准确性不可控，可能幻觉 |
| **双模混合** | 取长补短，灵活组合 | 架构复杂，维护成本增加 |

**最终决策：双模混合架构**

**实施策略：**
1. 有官方监管文档的场景 → RAG模式（SVF合规审查）
2. 无现成文档但LLM已有知识的场景 → LLM-only模式（其他3个）
3. 未来可添加文档 → 随时切换到RAG模式

#### 3.3.2 输出格式标准化

**问题：** LLM输出格式不稳定，影响用户体验

**我的解决方案：**

1. **超详细的格式Prompt**：在Prompt中明确指定每一级标题、表格结构
2. **格式后处理函数**：`format_output()`统一清理和标准化
3. **类型适配层**：`extract_response_content()`处理多种返回类型

### 7.4 我的风险决策案例

**案例1：LLM返回类型不兼容**

**问题：** RAG返回dict，LLM-only返回AIMessage，导致`format_output()`报错

**我的分析过程：**
1. 先看错误日志，定位问题
2. 分析两种模式的返回值差异
3. 设计通用的`extract_response_content()`函数
4. 测试验证所有场景

**最终方案：**
```python
def extract_response_content(response) -> str:
    """处理多种返回类型：str, dict, AIMessage等"""
    if isinstance(response, str):
        return response
    if isinstance(response, dict):
        return response.get('result', response.get('content', str(response)))
    if isinstance(response, AIMessage):
        return response.content
    # ... 更多类型处理
```
